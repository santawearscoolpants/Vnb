Supabase Auth emails (signup confirmation, password reset, magic link) via Resend
================================================================================

Supabase’s own mail is optional. Point Auth at Resend’s SMTP so all auth emails
go out through your verified domain (same provider you use for Edge Functions /
Worker if you like).

1) Resend
   - Verify your domain (DNS: SPF/DKIM as Resend shows).
   - Create an API key (re_...).

2) Supabase Dashboard → Project Settings → Authentication
   - Open the SMTP / “Custom SMTP” (or “Send emails”) section.
   - Enable custom SMTP and enter (from https://resend.com/docs/send-with-smtp):

       Host:     smtp.resend.com
       Port:     465 (SSL) or 587 (STARTTLS) — match what the form asks for
       Username: resend
       Password: YOUR_RESEND_API_KEY   (the re_... key)

   - Set “Sender email” to an address on your verified domain, e.g.
       orders@vnbway.com  or  noreply@vnbway.com
   - Set “Sender name” e.g. Vines & Branches

3) Save. Trigger a test: sign up a new user or use “Forgot password” and
   confirm mail appears in Resend → Emails and the inbox.

4) Email templates
   - Same section: Authentication → Email Templates — edit subjects/bodies if needed.

This does not replace your Edge Function `resend-email-hook` (database webhooks);
it only changes how Supabase Auth sends its own system emails.

VNB Steward application / waitlist emails (resend-email-hook)
-------------------------------------------------------------
These are **not** Auth emails. They fire from Database Webhooks → Edge Function
`supabase/functions/resend-email-hook` (Resend HTTP API).

Deploy the function and set secrets: RESEND_API_KEY, RESEND_FROM, WEBHOOK_SECRET,
SUPABASE_SERVICE_ROLE_KEY (see comments in the function file).

Then Dashboard → Database → Webhooks (or Integrations → Webhooks), add:

  1) Table `public.steward_applications`, event INSERT → function URL  
     — Sends “We received your VNB Steward application” to the user’s auth email  
       (Account → VNB Steward apply flow).

  2) Table `public.steward_waitlist`, event INSERT → same function URL  
     — Sends a waitlist thank-you only when `converted_user_id` is null  
       (public interest form). Rows created by `submit_steward_application` set  
       `converted_user_id`, so the hook skips (no duplicate with (1)).

Use header `x-webhook-secret` with the same value as WEBHOOK_SECRET.

If (1) is missing, signed-in applicants get **no** confirmation email even though
the app shows success. If migration `05_steward_applications.sql` was never applied,
`submit_steward_application` fails and nothing is stored — check SQL Editor for the
function and tables `steward_applications`, `steward_waitlist`.

Redirect after “Confirm email” (avoid localhost:3000)
-----------------------------------------------------
If the confirmation link opens http://localhost:3000, Supabase **Site URL** is still
set to dev. Fix:

  Dashboard → Authentication → URL Configuration
    • Site URL: https://www.vnbway.com   (your real storefront origin)
    • Additional Redirect URLs (add each on its own line, include wildcards if offered):
        https://www.vnbway.com/**
        https://vnbway.com/**
        http://localhost:3000/**          (optional, for local dev)

The storefront also sends emailRedirectTo on sign-up (current browser origin) so new
signups from production use the correct host even before you fix Site URL.

Optional: remove a stuck / unconfirmed user
--------------------------------------------
Prefer: Authentication → Users → select user → Delete.

If you use SQL Editor and your project allows it (watch for FKs to public.*):

  delete from auth.users where email = 'vnbwayltd@gmail.com';

See also: supabase/08_remove_test_auth_user.sql
